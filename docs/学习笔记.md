## 2026-06-18 - frpc TOML 编辑页面

- 主题：新增一个用于编辑 `config/frpc.toml` 的 GUI 页面。
- 当前理解：用户希望在 EasyFrp 内部直接编辑 frpc 的 TOML 配置，而不是手动打开文件。
- 架构方向：主窗口负责侧边栏和页面切换；新建 page 负责页面标题和整体布局；新建 panel 负责编辑器、按钮和保存/校验交互；配置文件读写逻辑优先放到 `core` 层，避免 UI 直接承担业务文件操作。
- 关键概念：页面切换使用 `QStackedWidget`；TOML 校验应使用 `tomlkit`，因为项目已经引入该依赖，并且它适合保留 TOML 的原始结构。
- 下一步：用户需要用自己的话复述页面、panel、core 三层分别负责什么，然后选择是否进入实现阶段。

### 用户复述后的修正

- 用户理解：`core service` 负责文件读取和修改；`panel` 组装编辑框和保存按钮，并调用 core 业务；`page` 导入 panel；`main_window` 注册新页面。
- 修正点：按钮点击不要直接绑定成“按钮 = core 保存方法”。更好的边界是：panel 读取编辑框内容，调用 core 校验/保存，再由 panel 决定如何显示错误、成功提示和状态栏消息。
- 待明确：这里的“编辑字段”第一版建议先理解为编辑完整 TOML 文本；如果要做字段化表单，需要额外设计 TOML 结构到表单控件的映射。

### 入门级实现记录

- 新增 `FrpcConfigService`：集中负责读取、校验、保存 `config/frpc.toml`。
- 新增 `FrpcTomlEditorPanel`：负责编辑框、重新加载按钮、校验按钮、保存按钮，以及成功/失败提示。
- 新增 `FrpcConfigView`：作为侧边栏中的完整页面，放置标题、说明和配置编辑 panel。
- 修改 `MainWindow`：侧边栏新增“frpc 配置”，并把新页面加入 `QStackedWidget`。
- 新概念：按钮点击信号先进入 panel 的处理函数，panel 再调用 core service；这能保持“UI 反馈”和“文件业务逻辑”分离。
## 2026-06-21 - frpc 连接配置模型设计

- 主题：准备在 `models` 层增加 frpc 连接配置模型，并把用户创建的连接配置持久化到 `data/profile/frpc`。
- 当前理解：用户希望按连接类型拆分数据类，例如 TCP、UDP 分别有自己的配置字段，后续再由这些模型生成或保存对应的 frpc TOML 配置。
- 架构方向：`models` 只表达数据结构和基础校验；文件路径和读写细节应放到 `core` 或专门的 profile repository/service；UI 后续只负责收集表单字段并调用服务，不直接拼 TOML 或写文件。
- 关键概念：`config/frpc.toml` 是当前正在运行的 frpc 配置文件；`data/profile/frpc` 更适合作为 EasyFrp 自己管理的连接配置档案目录，两者不要混成一个职责。
- 下一步：用户需要复述模型、持久化服务、现有 TOML 编辑服务三者的边界，并选择实现等级后再进入代码实现。
